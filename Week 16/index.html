🔑 Hashing Fundamentals

A hash table maps keys to array indices using a hash function. The goal is O(1) average time for insert, delete, and lookup operations.

Hash Function Requirements

  • Deterministic: same key always produces the same hash
  • Uniform: distributes keys evenly across buckets
  • Fast: cheap to compute

Collision Resolution

1. Chaining: Each bucket holds a linked list. Average time: O(1 + α) where α = load factor 2. Open Addressing: Find next open slot. - Linear Probing: check slot+1, slot+2, ... - Quadratic Probing: check slot+1, slot+4, slot+9, ... - Double Hashing: use a second hash function for step size

Java HashMap Internals

PropertyValue
Initial capacity16 buckets
Load factor threshold0.75
Resize strategyDouble buckets, rehash all entries
Collision handlingChaining (linked list → red-black tree when chain > 8)
Average timeO(1)
Worst-case timeO(n) (all keys hash to same bucket)
Key Insight
The load factor α = n/m (elements/buckets) controls the tradeoff between space and speed. Java's default threshold of 0.75 means the table resizes when it is 75% full, keeping operations near O(1).

⚙ Hash Function Design

A good hash function is the foundation of an efficient hash table. The goal is to map arbitrary keys to uniformly distributed integer indices with minimal collisions.

Properties of a Good Hash Function

  • Deterministic: The same input must always produce the same output
  • Uniform distribution: Outputs should be spread evenly across the hash space, minimizing clustering
  • Avalanche effect: A small change in input should produce a drastically different hash value
  • Efficient to compute: The function must run in O(1) time (constant relative to table size)
  • Minimize collisions: Different keys should map to different indices as often as possible
Key Insight
For integer keys, a common approach is h(k) = k mod m, where m is a prime number not close to a power of 2. For string keys, polynomial rolling hashes (like h = s[0]*p^(n-1) + s[1]*p^(n-2) + ... + s[n-1]) distribute well when p is a small prime (31 or 37).

Simple Hash Function Implementation

Below is a custom hash function for strings using polynomial rolling -- the same technique behind Java's String.hashCode().

HashFunctionDesign.java
// Polynomial rolling hash for strings static int customHash(String key, int tableSize) { int hash = 0; int prime = 31; for (int i = 0; i < key.length(); i++) { hash = hash * prime + key.charAt(i); hash %= tableSize; // keep within bounds } return Math.abs(hash) % tableSize; } // For integer keys -- multiplicative hashing (Knuth) static int intHash(int key, int tableSize) { // Knuth's multiplicative method long golden = 2654435769L; // 2^32 * (sqrt(5)-1)/2 long hash = (key * golden) >>> 16; return (int)(Math.abs(hash) % tableSize); }

hash_function.rs
/// Polynomial rolling hash for strings fn custom_hash(key: &str, table_size: usize) -> usize { let prime: usize = 31; let mut hash: usize = 0; for byte in key.bytes() { hash = hash.wrapping_mul(prime).wrapping_add(byte as usize); } hash % table_size } /// FNV-1a hash -- fast and well-distributed fn fnv1a_hash(key: &str) -> u64 { let mut hash: u64 = 14695981039346656037; for byte in key.bytes() { hash ^= byte as u64; hash = hash.wrapping_mul(1099511628211); } hash }
Common Pitfall
Never use a hash table size that is a power of 2 with a modular hash function unless you also mix the higher bits into the lower bits. Powers of 2 only use the lower bits of the hash, which increases collisions. Java's HashMap handles this internally by applying a secondary bit-spread function.

🔗 Collision Resolution: Chaining vs Open Addressing

When two keys hash to the same index, we have a collision. The two primary strategies for handling collisions have very different performance characteristics.

Chaining (Separate Chaining)

Each bucket holds a linked list (or other container) of all entries that hash to that index.

Bucket Array: ┌───┐ │ 0 │ → [("apple", 5)] → [("grape", 2)] → null ├───┤ │ 1 │ → [("banana", 7)] → null ├───┤ │ 2 │ → null (empty bucket) ├───┤ │ 3 │ → [("cherry", 1)] → [("date", 9)] → [("fig", 3)] → null ├───┤ │ 4 │ → null └───┘ Load factor α = n/m = 6/5 = 1.2 Average chain length ≈ α = 1.2

Chaining Implementation

ChainingHashMap.java
class ChainingHashMap<K, V> { private static class Node<K, V> { K key; V value; Node<K, V> next; Node(K k, V v, Node<K, V> n) { key = k; value = v; next = n; } } private Node<K, V>[] buckets; private int size; private static final double LOAD_THRESHOLD = 0.75; @SuppressWarnings("unchecked") ChainingHashMap(int capacity) { buckets = new Node[capacity]; } private int index(K key) { return (key.hashCode() & 0x7fffffff) % buckets.length; } void put(K key, V value) { int idx = index(key); for (Node<K, V> n = buckets[idx]; n != null; n = n.next) { if (n.key.equals(key)) { n.value = value; return; } } buckets[idx] = new Node<>(key, value, buckets[idx]); // prepend size++; if ((double) size / buckets.length > LOAD_THRESHOLD) resize(); } V get(K key) { int idx = index(key); for (Node<K, V> n = buckets[idx]; n != null; n = n.next) { if (n.key.equals(key)) return n.value; } return null; } @SuppressWarnings("unchecked") private void resize() { Node<K, V>[] old = buckets; buckets = new Node[old.length * 2]; size = 0; for (Node<K, V> head : old) for (Node<K, V> n = head; n != null; n = n.next) put(n.key, n.value); } }

Open Addressing (Linear Probing)

All entries live directly in the bucket array. On collision, probe the next slot until an empty one is found.

Insert "apple"(hash=3), "banana"(hash=1), "cherry"(hash=3), "date"(hash=3): Step 1: put "apple" at index 3 [ _ , _ , _ , apple, _ , _ , _ ] Step 2: put "banana" at index 1 [ _ , banana, _ , apple, _ , _ , _ ] Step 3: "cherry" hashes to 3 → occupied! probe 4 → empty [ _ , banana, _ , apple, cherry, _ , _ ] Step 4: "date" hashes to 3 → occupied! 4 → occupied! 5 → empty [ _ , banana, _ , apple, cherry, date, _ ] Clustering problem: consecutive filled slots slow down probing!

Open Addressing Implementation

OpenAddressingMap.java
class OpenAddressingMap<K, V> { private K[] keys; private V[] values; private boolean[] occupied; private int size, capacity; @SuppressWarnings("unchecked") OpenAddressingMap(int cap) { capacity = cap; keys = (K[]) new Object[cap]; values = (V[]) new Object[cap]; occupied = new boolean[cap]; } void put(K key, V value) { if (size * 2 >= capacity) resize(); int idx = (key.hashCode() & 0x7fffffff) % capacity; while (occupied[idx]) { if (keys[idx].equals(key)) { values[idx] = value; return; } idx = (idx + 1) % capacity; // linear probe } keys[idx] = key; values[idx] = value; occupied[idx] = true; size++; } V get(K key) { int idx = (key.hashCode() & 0x7fffffff) % capacity; while (occupied[idx]) { if (keys[idx].equals(key)) return values[idx]; idx = (idx + 1) % capacity; } return null; // not found } @SuppressWarnings("unchecked") private void resize() { K[] oldK = keys; V[] oldV = values; boolean[] oldO = occupied; capacity *= 2; size = 0; keys = (K[]) new Object[capacity]; values = (V[]) new Object[capacity]; occupied = new boolean[capacity]; for (int i = 0; i < oldK.length; i++) if (oldO[i]) put(oldK[i], oldV[i]); } }
open_addressing.cpp
template <typename K, typename V> class OpenAddressingMap { enum class State { EMPTY, OCCUPIED, DELETED }; struct Slot { K key; V value; State state = State::EMPTY; }; std::vector<Slot> table; size_t sz = 0; size_t probe(const K& key) const { size_t idx = std::hash<K>{}(key) % table.size(); while (table[idx].state == State::OCCUPIED && table[idx].key != key) idx = (idx + 1) % table.size(); return idx; } public: explicit OpenAddressingMap(size_t cap = 16) : table(cap) {} void put(const K& key, const V& value) { if (sz * 2 >= table.size()) resize(); size_t idx = probe(key); if (table[idx].state != State::OCCUPIED) ++sz; table[idx] = {key, value, State::OCCUPIED}; } V* get(const K& key) { size_t idx = probe(key); if (table[idx].state == State::OCCUPIED) return &table[idx].value; return nullptr; } void resize() { auto old = std::move(table); table.assign(old.size() * 2, Slot{}); sz = 0; for (auto& s : old) if (s.state == State::OCCUPIED) put(s.key, s.value); } };

open_addressing.rs
use std::collections::hash_map::DefaultHasher; use std::hash::{Hash, Hasher}; struct OpenAddressingMap<K, V> { slots: Vec<Option<(K, V)>>, size: usize, } impl<K: Hash + Eq + Clone, V: Clone> OpenAddressingMap<K, V> { fn new(capacity: usize) -> Self { Self { slots: (0..capacity).map(|_| None).collect(), size: 0 } } fn hash_index(&self, key: &K) -> usize { let mut h = DefaultHasher::new(); key.hash(&mut h); h.finish() as usize % self.slots.len() } fn put(&mut self, key: K, value: V) { if self.size * 2 >= self.slots.len() { self.resize(); } let mut idx = self.hash_index(&key); loop { match &self.slots[idx] { Some((k, _)) if *k == key => { self.slots[idx] = Some((key, value)); return; } None => { self.slots[idx] = Some((key, value)); self.size += 1; return; } _ => idx = (idx + 1) % self.slots.len(), } } } fn get(&self, key: &K) -> Option<&V> { let mut idx = self.hash_index(key); loop { match &self.slots[idx] { Some((k, v)) if k == key => return Some(v), None => return None, _ => idx = (idx + 1) % self.slots.len(), } } } fn resize(&mut self) { let old = std::mem::replace( &mut self.slots, (0..self.slots.len() * 2).map(|_| None).collect(), ); self.size = 0; for slot in old.into_iter().flatten() { self.put(slot.0, slot.1); } } }

Chaining vs Open Addressing Comparison

PropertyChainingOpen Addressing
Load factorCan exceed 1.0Must stay below 1.0 (ideally < 0.7)
Cache performancePoor -- pointer chasingGood -- contiguous memory
DeletionEasy -- remove from listTricky -- needs tombstone markers
ClusteringNo clusteringPrimary/secondary clustering
Memory overheadPointers per nodeEmpty slots (wasted space)
Used byJava HashMap, Go mapPython dict, Rust HashMap
Key Insight
Modern CPUs strongly favour cache-friendly data structures. Open addressing with linear probing often outperforms chaining in practice despite worse theoretical clustering, because sequential memory access is much faster than following linked-list pointers scattered in memory.

🔍 Two Sum

Given an array and a target, return indices of two numbers that sum to target. This is the quintessential HashMap problem.

Strategy

For each element, compute complement = target - nums[i]. Check if the complement is already in the map. If yes, we found the pair. Otherwise, store (nums[i], i) in the map.

Array: [2, 7, 11, 15], target = 9 i=0: complement = 9-2 = 7, map={}, not found → store {2:0} i=1: complement = 9-7 = 2, map={2:0}, found! → return [0, 1]

Implementation

HashingAndHashMap.java
static int[] twoSum(int[] nums, int target) { Map<Integer, Integer> map = new HashMap<>(); // value → index for (int i = 0; i < nums.length; i++) { int complement = target - nums[i]; if (map.containsKey(complement)) return new int[]{map.get(complement), i}; map.put(nums[i], i); } return new int[]{-1, -1}; // no solution }
hashing.cpp
std::optional<std::pair<int,int>> two_sum(const std::vector<int>& nums, int target) { std::unordered_map<int,int> seen; // value -> index for (int i = 0; i < static_cast<int>(nums.size()); ++i) { int complement = target - nums[i]; if (auto it = seen.find(complement); it != seen.end()) { return std::make_pair(it->second, i); } seen[nums[i]] = i; } return std::nullopt; // no solution }

📝 Group Anagrams

Group strings that are anagrams of each other. Two strings are anagrams if they have the same characters in the same frequency.

Strategy

Sort each string to produce a canonical key. Strings with the same sorted form are anagrams. Use this sorted string as a HashMap key.

Input: ["eat", "tea", "tan", "ate", "nat", "bat"] Sorted keys: "eat" → "aet" "tea" → "aet" "ate" → "aet" "tan" → "ant" "nat" → "ant" "bat" → "abt" Groups: [["eat","tea","ate"], ["tan","nat"], ["bat"]]
HashingAndHashMap.java
static List<List<String>> groupAnagrams(String[] strs) { Map<String, List<String>> map = new HashMap<>(); for (String s : strs) { char[] chars = s.toCharArray(); Arrays.sort(chars); // sorted key identifies anagram group String key = new String(chars); map.computeIfAbsent(key, k -> new ArrayList<>()).add(s); } return new ArrayList<>(map.values()); }
hashing.cpp
std::vector<std::vector<std::string>> group_anagrams(const std::vector<std::string>& strs) { std::unordered_map<std::string, std::vector<std::string>> groups; for (const auto& s : strs) { std::string key = s; std::sort(key.begin(), key.end()); // canonical form groups[key].push_back(s); } std::vector<std::vector<std::string>> result; for (auto& [key, group] : groups) result.push_back(std::move(group)); return result; }

🔢 Subarray Sum Equals k

Count the number of contiguous subarrays whose sum equals k. For each index j, we need to count how many earlier prefix sums equal prefix[j] - k.

Key Insight
Instead of checking all pairs (O(n^2)), maintain a running count of each prefix sum in a HashMap. For each new prefix sum, look up how many times prefix - k has appeared. This gives O(n) time.
HashingAndHashMap.java
static int subarraySum(int[] nums, int k) { Map<Integer, Integer> prefixCount = new HashMap<>(); prefixCount.put(0, 1); int sum = 0, count = 0; for (int x : nums) { sum += x; count += prefixCount.getOrDefault(sum - k, 0); prefixCount.merge(sum, 1, Integer::sum); } return count; }
hashing.cpp
int subarray_sum_equals_k(const std::vector<int>& nums, int k) { std::unordered_map<int,int> prefix_count; prefix_count[0] = 1; // empty prefix int sum = 0, count = 0; for (int x : nums) { sum += x; if (auto it = prefix_count.find(sum - k); it != prefix_count.end()) count += it->second; ++prefix_count[sum]; } return count; }
hashing.py
def subarray_sum_equals_k(nums: List[int], k: int) -> int: count = 0 prefix = 0 prefix_counts: Dict[int, int] = defaultdict(int) prefix_counts[0] = 1 # empty prefix for num in nums: prefix += num count += prefix_counts[prefix - k] prefix_counts[prefix] += 1 return count
hashing.rs

⚡ Longest Consecutive Sequence

Given an unsorted array, find the length of the longest sequence of consecutive integers. Example: [100,4,200,1,3,2] → 4 (sequence 1,2,3,4).

Strategy

Put all numbers in a HashSet for O(1) lookup. For each number, only start counting if num-1 is NOT in the set (i.e., it is the beginning of a sequence). Then extend forward.

Set: {100, 4, 200, 1, 3, 2} 100: 100-1=99 not in set → start! length=1 (just 100) 4: 4-1=3 IS in set → skip (not a sequence start) 200: 200-1=199 not in set → start! length=1 1: 1-1=0 not in set → start! extend: 2,3,4 → length=4 ✔ 3: 3-1=2 IS in set → skip 2: 2-1=1 IS in set → skip Answer: 4
Why O(n)?
Each element is visited at most twice: once as a potential start, and at most once while extending a sequence. The "only start from the beginning" check ensures no element is counted multiple times across different sequences.
HashingAndHashMap.java
static int longestConsecutive(int[] nums) { Set<Integer> set = new HashSet<>(); for (int x : nums) set.add(x); int longest = 0; for (int x : set) { if (!set.contains(x - 1)) { // only start from sequence beginning int curr = x, length = 1; while (set.contains(curr + 1)) { curr++; length++; } longest = Math.max(longest, length); } } return longest; }
hashing.cpp
int longest_consecutive_sequence(const std::vector<int>& nums) { std::unordered_set<int> num_set(nums.begin(), nums.end()); int best = 0; for (int n : num_set) { if (num_set.count(n - 1)) continue; // not a start int length = 1, cur = n; while (num_set.count(cur + 1)) { ++cur; ++length; } best = std::max(best, length); } return best; }
hashing.py

fn longest_consecutive_sequence(nums: &[i32]) -> i32 { let num_set: HashSet<i32> = nums.iter().copied().collect(); let mut best = 0; for &num in &num_set { if !num_set.contains(&(num - 1)) { let mut current = num; let mut length = 1; while num_set.contains(&(current + 1)) { current += 1; length += 1; } best = best.max(length); } } best }

Common Pitfall
Hash table operations are O(1) on average. In the worst case (all keys hash to the same bucket), they degrade to O(n). In competitive programming, adversarial inputs can trigger this. Use randomized hashing or switch to balanced BST maps (TreeMap) if needed.
